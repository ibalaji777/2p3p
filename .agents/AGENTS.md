# Material Independence Rule for Attached Elevation Elements

**WARNING**

Any elevation element attached to a wall (such as sheds, chajjas, cornices, moldings, fascias, balconies, columns, trims, decorative panels, window frames, door frames, arches, or any future component) must never interfere with the wall's material system.

The material system and the geometry generation system must remain completely independent.

## Required Behavior
* Attaching an elevation component must not change the wall's material.
* Removing an attached component must not reset or alter the wall's material.
* Regenerating wall geometry must preserve all existing material assignments.
* Newly generated wall faces must inherit the correct material instead of falling back to the default material.
* Every elevation component should manage only its own materials.
* Wall materials should always be controlled exclusively by the wall material system.

## Design Principle
Geometry modifies shape. Materials modify appearance. These systems must never depend on each other.

Any attached elevation feature should behave as an independent object that contributes geometry only, without modifying, resetting, or replacing the material assignments of the parent wall.

## Verification
After implementing any new elevation feature, verify that:
1. Wall materials remain unchanged after adding the feature.
2. Wall materials remain unchanged after editing the feature.
3. Wall materials remain unchanged after deleting the feature.
4. Multiple attached elements can coexist without creating unpainted (default white) regions.
5. Undo/Redo preserves both geometry and material assignments independently.

This rule should apply universally to all current and future elevation components to ensure a robust, scalable rendering architecture.

# Widget Material Updates & UV Scaling Rule

**WARNING**

When applying materials to 3D widgets (like elevation_fascia, sunshade, or moldings) via the Material Gizmo (GizmoManager.js), NEVER replace the materials directly in-place on the existing meshes using 	raverse or array assignments.

You MUST rely on updateMaterialLive(entity) to completely rebuild the widget through its ender3D registry function.

## Why this is required
1. **UV Scaling:** Widget geometries (like ExtrudeGeometry) often rely on getFaceMaterials to calculate the correct UV scaling factors (epeat.set(...)) based on the component's exact physical dimensions.
2. **Texture Tiling:** If a material is injected in-place without rebuilding the widget, the texture repeat scaling is lost. This causes patterns like brick or wood to stretch into a single massive pixelated tile across the entire object (resulting in the "box shape" bug).

## Required Behavior
* In engine3d.GizmoManager.js, always use if (this.ctx.updateMaterialLive) this.ctx.updateMaterialLive(entity); after updating entity.params for widgets.
* If a widget component (like sunshade) needs multi-face material inheritance, that logic must be written inside its ender3D function in egistry.js, so that it correctly delegates to getFaceMaterials when rebuilt.
* In-place material swapping (mesh.material[x] = newMat) is ONLY permitted for Base Walls (entity.type === 'outer' or 'inner'), because walls calculate their UV projections globally on the vertex level inside EnvironmentBuilder.js.


# Three.js ExtrudeGeometry & Material Index Mapping Rule

**WARNING**

When constructing 3D meshes using `THREE.ExtrudeGeometry` (e.g., carved door panels, moldings), be extremely careful with how material indices map to the geometry faces.

## Why this is required
Unlike `THREE.BoxGeometry` which expects an array of 6 materials mapped sequentially to its 6 sides (Right, Left, Top, Bottom, Front, Back), `THREE.ExtrudeGeometry` only uses **two** material indices:
- `materialIndex 0`: Applied to the front and back flat caps (the shape itself).
- `materialIndex 1`: Applied to the extruded/beveled sides.

If you pass a standard 6-material array (generated by `getFaceMaterials().box`) into an `ExtrudeGeometry` mesh, index 0 (Right side) will mistakenly be applied to the front face of the geometry!

## Required Behavior
1. **Material Array Re-mapping**: When passing materials to an `ExtrudeGeometry` mesh in `registry.js`, you must construct a specialized array. For example, if you want the front texture (`mats[4]`) on the face and the side texture (`mats[1]`) on the bevels:
   ```javascript
   const matsExtrude = Array.isArray(mats) ? [mats[4], mats[1]] : mats;
   const mesh = new THREE.Mesh(geo, matsExtrude);
   ```
2. **GizmoManager SubMesh Logic**: For monolithic entities like doors that are painted as a whole, do NOT use `entity.params.blocks[subMeshIndex]` inside `GizmoManager.js`. Raycasting a specific child panel mesh will return an `activeSubMeshIndex`, but doors should route material assignments directly to `entity.params.textureFront` (or the respective face parameter) rather than nested sub-blocks. 
   ```javascript
   if (this.activeSubMeshIndex !== -1 && !entity.type.startsWith('shape_') && entity.type !== 'door') {
       // Only use blocks for objects that actually support independent sub-materials
   }
   ```

# Wall Hole Shearing & Miter Joints Rule

**WARNING**

When applying a shear transformation to wall geometry to form mitered joints (e.g., `shearGeo` in `EnvironmentBuilder.js` or `Wall3DBuilder.js`), you MUST NOT shear the entire wall linearly.

## Why this is required
If the entire wall is sheared (shifted based on `x / length`), any holes (for windows, doors, etc.) inside the wall will also be sheared. This causes two critical bugs:
1. **Slanting:** Rectangular holes turn into parallelograms.
2. **Scaling:** The width of the hole is scaled proportionally to the corner stretch, making the hole wider or narrower than the rigid 3D frame that goes inside it. This guarantees gaps between frames and walls.

## Required Behavior
* The `shearGeo` function must ONLY shift the vertices at the extreme ends of the wall (`x <= 0.1` and `x >= length - 0.1`). 
* All internal vertices (holes, cutouts) must be left untouched (`pos.setX(i, x)`).
* Window and door meshes should simply be placed at `entity.localX = wCenter`, ensuring they match the mathematically perfect, un-sheared holes.

# Legacy Material Fallbacks & Registries Rule

**WARNING**

Do not attempt to manually inject legacy fallbacks for materials (like `helpers.getDoorMaterial`) or manually check undefined registries (like `DOOR_MATERIALS_REGISTRY`).

## Required Behavior
* Always use `helpers.getDynamicMaterial(matKey, type)` to fetch materials for doors, windows, and widgets. 
* `getDynamicMaterial` natively handles standard lookups, default fallbacks, and caching. Writing custom fallback blocks around it introduces `TypeError` crashes and circumvents the unified material pipeline.
